# Lodge Companion App — Build Plan

> Flutter mobile app serving **guests** (in-stay experience) and **cleaners** (task operations)
> under one codebase. Sits between the existing Website (pre-arrival) and Backoffice (internal ops).

---

## Decisions & Constraints

| Decision | Choice | Reason |
|---|---|---|
| State management | Riverpod + code gen | Clean async handling, role-based provider families, no boilerplate |
| Routing | go_router | Declarative, auth redirect guards built-in |
| Models | freezed + json_serializable | Immutable, copyWith, JSON serialization in one pass |
| HTTP client | dio | Interceptors for JWT injection |
| JWT storage | flutter_secure_storage | Plaintext SharedPreferences is not secure |
| Env config | flutter_dotenv | `.env` file for dev/prod base URLs |
| Real-time | None needed | Request status updates delivered by email to guest |
| Offline support | Not supported | App requires internet — good error states only |
| Invoice download | Not in scope | Hard copy only, generated by staff in backoffice |
| PDF handling | Not in scope | Removed from guest dashboard quick actions |
| Notifications | Phase 6 (FCM) | Checkout alerts, new assignments for cleaners |

## API Endpoints

All endpoints to be built on the backend. Base URL configured via `.env`.

### Auth
| Method | Endpoint | Used by |
|---|---|---|
| `POST` | `/auth/login` | Both |
| `POST` | `/auth/logout` | Both |

### Guest
| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/guest/booking` | Active booking + room info for dashboard |
| `POST` | `/guest/requests` | Submit a room service request |
| `GET` | `/guest/requests` | List own requests (load on screen open) |
| `PATCH` | `/guest/requests/:id` | Cancel a request |
| `POST` | `/guest/ratings` | Submit stay rating after checkout |

### Cleaner
| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/cleaner/rooms` | Today's assigned rooms ordered by priority |
| `POST` | `/rooms/:id/checklist` | Submit completed cleaning checklist |
| `PATCH` | `/rooms/:id/status` | Mark room as clean → status: `available` |
| `GET` | `/cleaning/history` | Own cleaning log (Phase 5) |
| `POST` | `/supply-requests` | Flag low supplies (Phase 5) |

### Admin (Backoffice — not built in this app)
| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/admin/requests` | View all guest requests |
| `PATCH` | `/admin/requests/:id/status` | Update request status |
| `GET` | `/admin/ratings` | View all stay ratings |
| `GET` | `/admin/cleaning/history` | View all cleaner logs |
| `GET` | `/admin/supply-requests` | View supply requests |

## Phased Build Plan

### Phase 0 — Project Foundation
> Clean structure, design system, shared infrastructure

- [ ] Replace default counter app — clean `main.dart`, proper `MyApp` shell
- [ ] Add all packages to `pubspec.yaml`
- [ ] Create `.env` + `.env.example` with `BASE_URL` variable
- [ ] Wire `flutter_dotenv` in `main.dart` before `runApp`
- [ ] Build `AppTheme` — full `ThemeData` from design tokens:
  - Color scheme (primary `#502c17`, secondary `#476363`, surface `#fff8f1`, tertiary `#38352f`)
  - Typography (Noto Serif for headlines, Manrope for body/labels)
  - Border radii (default `0.25rem`, lg `0.5rem`, xl `0.75rem`)
  - No dividers, no border rules enforced at theme level
- [ ] Build `ApiClient` — Dio instance with base URL + auth interceptor
- [ ] Build `SecureStorageService` — wrapper for JWT read/write/delete
- [ ] Build shared widgets: `TimberAppBar`, `TimberBottomNav`, `SurfaceCard`, `PrimaryButton`, `StatusBadge`

**Exit criteria:** App launches to a placeholder screen. Design system tokens verified visually.

---

### Phase 1 — Auth & Role Gate
> Login screen → role-based routing

- [ ] `User` model (freezed) — id, name, role (`guest` | `cleaner`), room number
- [ ] `AuthRepository` — login, logout, read stored session
- [ ] `AuthNotifier` (Riverpod `AsyncNotifier`) — wraps repository, exposes auth state
- [ ] `LoginScreen` — Noto Serif headline, Manrope inputs, borderless field style, primary button
- [ ] `go_router` config — auth redirect: unauthenticated → `/login`, authenticated → role-based shell
- [ ] Guest shell route (`/guest/...`) with `TimberBottomNav` (Stay | Dining | Tasks | Concierge)
- [ ] Cleaner shell route (`/cleaner/...`) with `TimberBottomNav` (Stay | Dining | Tasks | Concierge)
- [ ] Logout clears secure storage + redirects to login

**Exit criteria:** Login with cleaner credentials → cleaner shell. Login with guest credentials → guest shell.

---

### Phase 2 — Guest Dashboard + Room Service Requests
> Core guest in-stay experience (highest priority)

**2a — Guest Dashboard**
- [ ] `BookingRepository` — `GET /guest/booking`
- [ ] `GuestDashboardScreen`:
  - Personalised greeting (name + room number)
  - Booking card — room image (`cached_network_image`), room name, dates, meal plan
  - Quick actions: Make a Request → `/guest/requests`, Rate Your Stay → `/guest/rating`
  - Contextual suggestion card (static for now)
  - Shimmer skeleton while loading

**2b — Room Service Requests**
- [ ] `Request` model (freezed) — id, category, description, status, created_at, room
- [ ] `RequestsRepository` — `POST /guest/requests`, `GET /guest/requests`
- [ ] `RequestsScreen`:
  - Category selector grid (Housekeeping / Maintenance / Food & Beverage / Concierge)
  - Free-text textarea (borderless, bottom-border focus style)
  - Submit button → POST → optimistic list update
  - Active requests list — status badges (Pending amber / In Progress blue / Resolved green)
  - Pull-to-refresh on requests list

**Exit criteria:** Guest can submit a request and see it appear in their active list with correct status badge.

---

### Phase 3 — Cleaner Dashboard + Cleaning Checklist
> Core operational loop for cleaning staff

**3a — Cleaner Dashboard**
- [ ] `CleanerRoomRepository` — `GET /cleaner/rooms`
- [ ] `CleanerDashboardScreen`:
  - Shift greeting + live clock (`Timer.periodic` updating every second)
  - Stats bento grid — Assigned / Needs Cleaning / Completed (derived from room list)
  - Priority room list ordered: Checkout (red left border) → Not Ready (grey) → Stay-over (green)
  - Status badge per room: `CHECKOUT` / `NOT READY` / `STAY OVER`
  - Disabled chevron button for `not_ready` rooms (guest still in-room)

**3b — Cleaning Checklist**
- [ ] `ChecklistItem` model (freezed) — id, label, icon, isChecked
- [ ] Checklist varies by room type — map defined in app (not API-driven initially)
- [ ] `ChecklistScreen`:
  - Room name + type header
  - Guest special notes panel (left border accent)
  - Progress bar (X of Y tasks complete)
  - Checklist items — tap to toggle, checked items get strikethrough
  - "Mark as Clean" button — disabled until all items checked
  - On submit: `POST /rooms/:id/checklist` then `PATCH /rooms/:id/status`
  - Navigate back to dashboard on success, room removed from list

**Exit criteria:** Cleaner can work through a full room checklist and mark it clean. Room disappears from dashboard.

---

### Phase 4 — Stay Rating
> Close the guest feedback loop

- [ ] `Rating` model (freezed) — overall, roomCondition, staffFriendliness, mealQuality, cleanerRatings[], comment
- [ ] `RatingsRepository` — `POST /guest/ratings`
- [ ] `StayRatingScreen`:
  - Large overall star rating (1–5)
  - Secondary rating cards: Room Condition, Staff Friendliness, Meal Quality (if meal plan active)
  - Optional cleaner rating section — avatar + name + star row per assigned cleaner
  - Free-text comment textarea
  - Submit button → POST → success state
- [ ] `StarRatingRow` shared widget — reusable across all rating cards

**Exit criteria:** Guest can submit a complete rating. Submission confirmed with success UI.

---

### Phase 5 — Supply Requests + Cleaning History
> Complete the cleaner side

- [ ] `SupplyRequestScreen`:
  - Category chips: Towels / Toiletries / Linen / Minibar / Cleaning Products
  - Optional notes field
  - `POST /supply-requests`

- [ ] `CleaningHistoryScreen`:
  - Grouped list: Today / This Week
  - Per entry: room name, time completed, checklist snapshot (items ticked)
  - `GET /cleaning/history`

**Exit criteria:** Cleaners can flag supplies and review their own shift history.

---

### Phase 6 — Notifications + Polish
> Proactive alerts and final UX refinement

- [ ] Firebase project setup + FCM integration
- [ ] Push notification triggers:
  - Cleaner: guest checks out → room ready to clean
  - Cleaner: new room assignment
  - Cleaner: guest submits housekeeping request for their room
- [ ] `flutter_local_notifications` — show alerts when app is in foreground
- [ ] Skeleton loaders on all screens (using `shimmer`, matching surface tonal layers)
- [ ] Empty states — on-brand copy, not generic Material defaults
- [ ] Network error states — retry button, warm error messaging
- [ ] Accessibility audit — 44×44px touch targets, contrast ratios, semantic labels

**Exit criteria:** End-to-end flow works for both roles with no rough edges.

## Design System Reference

| Token | Value | Use |
|---|---|---|
| `primary` | `#502c17` | Branding, primary actions, headlines |
| `secondary` | `#476363` | Secondary actions, supporting UI |
| `surface` | `#fff8f1` | Page background |
| `surface-container-low` | `#fbf3e5` | Section backgrounds |
| `surface-container-lowest` | `#ffffff` | Interactive cards |
| `on-surface` | `#1f1b13` | All "black" text — never use `#000000` |
| `tertiary` | `#38352f` | Charcoal for grounding elements |
| `error` | `#ba1a1a` | Checkout urgency badges |

**Typography:**
- Headlines / display: Noto Serif
- Body / labels / buttons: Manrope
- Line height body: minimum 1.5

**Rules:**
- No 1px solid borders for sectioning — use background shifts and whitespace
- No sharp corners — minimum `BorderRadius.circular(4)` everywhere
- Bottom nav: floating glassmorphism, `BackdropFilter`, `BorderRadius.circular(16)`, 16px from screen edges
- Cards: `BorderRadius.circular(12)`, `surface-container-lowest` on `surface-container` background

- Invoice PDF download — hard copy only, generated by backoffice staff
- Offline mode — internet required, good error states only
- Guest booking cancellation — website handles pre-arrival cancellations
- Guest profile editing — not in current scope
- Admin/backoffice features — separate Vue 3 app
